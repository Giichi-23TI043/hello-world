# ソフトウェア工学の内容のまとめ

## ソフトウェア工学概論
ソフトウェア工学の知識の乖離は開発に支障をきたすため、チームで共有する必要がある。
ソフトウェアのシステムは常に環境やニーズを満たすように適応するために改良を日々続けていく必要がある。しかし現在、ソフトウェアの複雑化、大規模化、多様化などが進む一方でIT人材の不足及び知識不足が課題となっており、技術者の成長が目指されている。

## ソフトウェアライフサイクル
ソフトウェアを作るにはまず現在のユーザのニーズを把握し、要件を詳しく定義して実装すべき機能を明確にし、企画書及びWBSなどの仕様書などを作り、それらに従って、場合によっては外部に委託などをしてシステムを実際に構築していく。また、バグを早期に発見及び対処することが理想であり、事前にテストの範囲などを決めておく必要がある。稼働後も保守作業を怠らないことも大切である。

## プロジェクト
有期性、独自性が特徴のプロジェクトではバックキャスティングを採用しており、要件定義から実現する項目を設定し、それに向けて道筋を適宜検討する。

## ソフトウェア分析
ソフトウェアの評価では、品質、コスト、納期の観点でのシステムの優先順位を決め、コードの物量やバグの量、FP法、使い勝手などを考えて分析する。

## 開発プロセス
ウォーターフォール型開発プロセスでは上流から順に作業をこなすため管理が簡単であるが終盤でのリスクが高い。
スパイラルモデルではプロトタイプを作り、そのフィードバックにより開発を進める方法であるが、プロトタイプ作成が難航する恐れがある。
反復型開発プロセスでは部分的に段々と実装する方法だが、作業量の増加、全体像の不明瞭などの欠点がある。
アジャイル開発ではイテレーションという短い期間で反復開発を行うことで変化に柔軟に対応するといった方法で開発を進めるものであり、スクラム、カンバン、XPなどの手法がある。また、想定されるリスクをあらかじめ考えておくリスク駆動型開発プロセスの考えも反復開発に欠かせない手法である。

## WBS
WBSとはWork Breakdown Structureの略称であり、行うべき作業の分解、階層化を行うものである。これによりスコープが明確になり、これに則ってプロジェクトを進めることで目標が達成されるというものである。

# ソフトウェア工学の内容のまとめ(講義第7回～第13回追加分)

## コーディング
コーディングとは、ソフトウェア開発において重要なコードの可読性を向上させるための方法である。例えばPythonのPEP8において、コードは一行79文字以内、説明書きやコメントは72文字以内にすることで、可読性を上げている。ほかにも、行の継続された文では要素を縦にそろえる、インデント、スペースの使用、改行のルールなどがある。

## バージョン管理・git
チームでの開発において、変更点を管理することはとても重要である。作業を各々ローカルレポジトリで行い、達成すべき変更が完成したらリモートレポジトリに置くといった分散管理型が主なチーム開発における方法である。githubがもっとも有名な開発環境であり、Codespacesでブラウザ上でVScode環境を簡単に構築でき、マークダウンによって簡単に文章を記述することができる。それらの編集したものをコミット、プッシュ、プルリクエスト、マージなどの流れに沿って処理を行うことで変更、編集などが行える。
また、flake8という機能ではコードをチェックして、自動的に改善点を教えてくれる便利なものである。さらにGithub Pagesではリモートレポジトリの内容をwebpageとして簡単に公開できるようになっている。
